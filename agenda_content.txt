Visão Geral do ProdutoAgendaIA é uma plataforma SaaS multi-tenant que automatiza o agendamento de serviços para negócios como manicures, barbearias, clínicas, pet shops, estéticas e academias — tudo pelo WhatsApp, sem que o cliente precise baixar nenhum app ou acessar qualquer link.O cliente simplesmente manda mensagem. Um agente de IA, configurado com a identidade do estabelecimento, conduz toda a conversa: entende o pedido, verifica horários disponíveis em tempo real, confirma o agendamento e dispara lembretes automáticos.ProdutoSaaS B2B com assinatura mensal por tenantCanal principalWhatsApp via WuzAPI (self-hosted ou cloud)Automaçãon8n orquestra todos os fluxos e integraçõesMotor de IALLM (GPT-4o ou Claude 3.5) com contexto dinâmico por tenantFrontendNext.js 14 com App Router — painel do estabelecimento e super adminBanco de dadosTurso (SQLite distribuído) — leve, edge-ready, multi-tenantPúblico-alvoPMEs de serviços que dependem de agendamentos recorrentes2. Arquitetura Técnica2.1 Visão Geral da StackA arquitetura foi escolhida para máxima velocidade de desenvolvimento, custo operacional baixo e escalabilidade progressiva. Cada camada tem responsabilidade clara:WuzAPIGateway WhatsApp self-hosted baseado no Baileys. Recebe e envia mensagens, expõe webhook para o n8n. Cada tenant pode ter seu próprio número vinculado.n8nMotor de orquestração low-code. Contém todos os fluxos: recebimento de mensagem, chamada à IA, gravação no banco, envio de resposta, lembretes agendados e pós-atendimento.LLMGPT-4o ou Claude 3.5 via API. Recebe o system prompt montado dinamicamente com contexto do tenant e histórico da conversa. Retorna texto + ação estruturada em JSON.TursoSQLite distribuído na edge. Banco principal da plataforma. Isolamento multi-tenant por database separado por tenant (Turso suporta milhares de databases). Zero cold start.Next.js 14App Router com Server Components. Painel do estabelecimento (agenda, configurações, relatórios) e super admin. API routes para integrações internas.RedisGerenciamento de sessão de conversa por número de telefone. TTL de 4 horas por sessão inativa.2.2 Fluxo Completo de uma MensagemCLIENTE envia mensagem no WhatsApp   │   ▼WUZAPI recebe a mensagem e dispara webhook POST   → Payload: { from, body, timestamp, tenantId }   │   ▼N8N — Workflow &apos;Receber Mensagem&apos;   1. Identifica tenant pelo número de destino   2. Busca/cria sessão no Redis (key: tenant:telefone)   3. Carrega contexto do Turso:      - Serviços e preços do tenant      - Profissionais e disponibilidade      - Slots livres (próximos 7 dias)      - Histórico do cliente (por telefone)   4. Monta system prompt com variáveis preenchidas   5. Chama LLM (GPT-4o / Claude) com histórico   │   ▼LLM retorna JSON estruturado:   { mensagem: &apos;...&apos;, acao: &apos;BOOK|CANCEL|NONE&apos;, dados: {...} }   │   ▼N8N — executa ação (se houver):   BOOK   → INSERT agendamento no Turso + bloqueia slot   CANCEL → UPDATE status=&apos;cancelado&apos; + libera slot   NONE   → apenas responde   │   ▼WUZAPI envia resposta ao cliente   │   ▼N8N atualiza sessão no Redis + agenda lembretes no BullMQ2.3 Workflows n8nCada funcionalidade é um workflow independente no n8n, o que permite manutenção e evolução sem risco de quebrar outros fluxos:Receber mensagem WhatsApp → Webhook WuzAPI → Redis → Turso → LLM → respostaLembrete 24h antes → Cron diário → busca agendamentos do dia seguinte no Turso → WuzAPI envia lembrete personalizadoLembrete 1h antes → Cron horário → busca agendamentos da próxima hora → WuzAPI envia com endereço e orientaçõesAvaliação pós-atendimento → Cron 30min após fim do atendimento → WuzAPI envia NPS → aguarda resposta → grava no TursoReativação de clientes inativos → Cron semanal → query Turso por clientes sem agendamento há X dias → WuzAPI envia mensagem de reativaçãoNotificação do estabelecimento → Após cada agendamento confirmado → WuzAPI notifica número do dono com resumo2.4 Modelo de Dados — Turso (SQLite)Cada tenant possui seu próprio database no Turso.
